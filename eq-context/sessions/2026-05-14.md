## RLS flag-back

Per `docs/cowork-prompt-v3.md` §"Visibility (draft mode)":
- While `tenders.stage in (watch, likely, won)`: nominations visible to managers ONLY.
- When `tenders.stage = 'confirmed'`: nominations visible to the nominee.
- Implementation note in the spec: "Implement via RLS policies on `nominations` and `pending_schedule`. Flag if auth context isn't sufficient to drive the policy."

eq-solves-field auth is **PIN-based** via the `/.netlify/functions/verify-pin` endpoint. The function signs a 7-day token, but Supabase calls go via the public anon key — not a user-scoped JWT. There's no `auth.uid()` in the request, so RLS can't distinguish "manager" from "nominee" from "anyone".

What landed instead: anon-role permissive policies matching the existing people / sites / schedule shape. Org-isolation is enforced **client-side** by `sbFetch` auto-appending `org_id=eq.<TENANT.ORG_UUID>` (per `ORG_TABLES` in app-state.js).

To do the spec properly, eq-solves-field needs:
- Real per-user Supabase auth (email magic-link, OIDC, or a Netlify→Supabase JWT exchange).
- RLS policies on `nominations` / `pending_schedule` checking the JWT user's role + person_id.
- Frontend rewrite to pass the user JWT through to all Supabase calls instead of the anon key.

This is a bigger conversation than the pipeline module. Calling it out so the gap is explicit, not hidden behind permissive policies.

## Cowork sandbox limitations encountered

1. **No git on the Windows mount mid-session.** Initial plan was to emit a .bat for Royce. Mid-way through, Royce supplied .git-credentials and I switched to pushing directly from a sandbox-side `git clone` in /tmp. Mount remained read-only to git operations because Windows held .git/HEAD.lock open.

2. **Stale bash mount on Edit-tool modifications.** The Linux sandbox mount of the Windows workspace doesn't refresh promptly when files are modified via the Edit tool (in-place edits to existing files). Files created via the Write tool appear in the mount immediately, but Edited files retain their pre-edit view until something invalidates the cache. Concrete symptom: `node --check scripts/app-state.js` reported "Unexpected end of input" against a truncated cached view (471 lines, stops mid-SEED array) while the Read tool confirmed the live file ends correctly at line 482 with `};`. The deployed file via Netlify will be the live Windows-side version — the sandbox view is misleading but the on-disk file is fine. Workaround when verification matters: re-read via Read tool or copy the file to `/tmp` before running node-based checks.

3. **Transactions in Supabase MCP.** `execute_sql` doesn't honour explicit `BEGIN`/`COMMIT` — the first attempt at the seed cleanup rolled back silently because the MCP wrapped each statement in its own implicit transaction. Workaround: run DELETE / INSERT as separate statements.
