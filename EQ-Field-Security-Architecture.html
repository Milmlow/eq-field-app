How It Works — In Everyday Terms

🛂
Login Security
Technical term: PIN-based access + session tokens
Like a site access code. Everyone uses the same code (just like a workplace gate), but the system tracks who's logged in and from where. If you close the app, you need to enter the code again to get back in.
🏨
Company Separation
Technical term: Multi-tenant Row-Level Security
Think of a secure filing cabinet. SKS has its own drawer with its own lock. Even if someone broke into the cabinet, they still couldn't open the SKS drawer without the right key.
📸
Audit Trail
Technical term: Immutable change logging
Every change is recorded like security camera footage — who made it, when, and what changed. These records can't be edited or deleted. Perfect for payroll disputes or compliance checks.
🔒
Data Encryption
Technical term: AES-256 at rest, TLS 1.2+ in transit
Your data is scrambled while it's stored and while it travels between devices and the server. Even if someone intercepted it, they'd see gibberish — like a letter in a code only you and the server can read.
☁️
Cloud Infrastructure
Technical term: Serverless cloud on Netlify + Supabase
There's no computer in an office running this app. It runs on enterprise cloud infrastructure (same as banks use) that automatically patches security updates and backs up your data every day.
📲
Progressive Web App
Technical term: Offline-capable PWA
The app works on any device — laptop, tablet, phone — and can work offline. When you reconnect, all your data syncs to the server automatically. No app store required.

Common Questions

Is the PIN shared? Isn't that less secure?
Yes, staff share a PIN (like a site access code), and supervisors have a separate PIN. This is appropriate for a field app where users are contractors using company devices. The system tracks which person logged in based on name selection, so you have full audit history. If someone's device is lost, a supervisor can remotely deactivate that person's access.
Can another company see SKS data?
No. Data separation is enforced by the database itself, not just the application. Even if there were a bug, the database would still block cross-company access. Two independent security layers would both need to fail simultaneously.
What happens if someone's phone is lost?
A supervisor can instantly deactivate that person's account. The very next time that device tries to use the app, the system signs them out automatically. The thief would also need the PIN to log back in.
Where is the data physically stored?
Sydney, Australia (AWS ap-southeast-2 region). Both the application server and the database are hosted in the same region. Your data doesn't leave the country.
Is our data backed up?
Yes. The database is backed up automatically every day by Supabase. Backups are encrypted. When you delete something in the app, it's hidden but not destroyed, so it can always be recovered.
Who can see what?
Staff can see their own schedule and limited details. Supervisors can see everything — rosters, timesheets, people management. Agency users see timesheets only. Nobody sees other companies' data.

Executive Summary

EQ Solves — Field is a workforce rostering and timesheet Progressive Web App (PWA) built for SKS Technologies electrical contractors managing schedules, hours, and leave across multiple sites.

The Field app is designed for rapid deployment and offline-capable operation. It uses PIN-based access (not per-user accounts), 7-day session tokens, and Supabase as the backend. Security is implemented through multi-tenant data isolation (row-level security), audit logging of all changes, rate limiting on authentication attempts, and encrypted transport.

As of v1.1, all CRITICAL security items are resolved: server-side secrets are injected via environment variables with no fallbacks in code, all endpoints are authenticated, CORS is whitelisted to approved origins, XSS input sanitisation is in place, HSTS enforces HTTPS, email inputs are validated, and all AI agent usage is audit-logged. A remediation roadmap tracks remaining HIGH and MEDIUM hardening items as the platform evolves toward production grade.

Infrastructure Overview

Hosting

The application is deployed on Netlify (Sydney region, ap-southeast-2) as a static PWA with serverless functions for PIN verification, API proxying, and email. There are no long-running servers to patch or maintain. Each HTTP request runs in an isolated function instance that terminates after responding.

Database

All data is stored in Supabase (managed PostgreSQL, Sydney region). Supabase provides the database, storage, and real-time subscriptions. Supabase is SOC 2 Type II certified and encrypts all data at rest (AES-256) and in transit (TLS 1.2+).

Multi-Tenant Isolation

EQ Solves Field is a multi-tenant platform. Each tenant (company) has complete data isolation enforced at the database level through PostgreSQL Row-Level Security (RLS).

How RLS Works

Every table in the database has RLS policies enabled. When a user queries any table, PostgreSQL automatically appends a filter clause that restricts rows to the user's tenant (identified via hostname slug mapping). This happens inside the database engine itself — even a bug in application code cannot expose another tenant's data.

-- Example RLS policy on timesheets table
CREATE POLICY tenant_isolation ON timesheets
  USING (org_id = CURRENT_SETTING('app.org_id'));

Tenant Boundary Enforcement

LayerMechanismBypass Risk
DatabaseRLS policies on every data tableNone — enforced at PostgreSQL level
ApplicationHostname → org_id mapping on every requestLow (RLS backstop active)
StorageRLS on Supabase Storage bucketsNone

Defence in depth: The application layer derives org_id from the request hostname and passes it to Supabase. Even if the application logic fails, the database RLS layer independently blocks cross-tenant queries. Both layers must fail simultaneously for data to leak.

Authentication & Access Control

PIN-Based Login Model

The Field app uses shared PINs, not per-user credentials. This is appropriate for field teams using company devices. There are two PIN codes:

CodeValue (Example)Who Uses ItPermissions
Staff PIN2026All staff membersView own schedule, record hours
Supervisor PINSKSNSWSupervisors onlyFull roster, timesheets, people management

Current state: PINs are currently shared across all staff of the same role. Per-user credentials are planned for v2.0. This is acceptable for v1.0 given that users are known contractors and devices are managed company equipment. Accountability is maintained via audit logging (every change is logged with timestamps and name selection).

Access Control Flow

User enters PIN → Verify against hash → Generate session token → Store locally (PWA) → Validate on requests

No MFA: The Field app does not implement multi-factor authentication. This is a known limitation documented in the remediation roadmap. For v1.0, rate limiting on PIN attempts and 7-day session expiry provide rate-limiting protections. MFA integration is planned for v2.0.

Data Retention & Deletion

The Field app uses soft deletes for all records. No data is permanently destroyed through the application — all records use an is_active flag for logical deletion. This preserves full audit history and allows recovery if needed.

Australian Privacy Principles Compliance

The Field app handles personal information (employee names, phone numbers, email addresses) and complies with the Australian Privacy Principles:

  • APP 1 (Open and transparent management): Privacy notice shown at login.
  • APP 3 (Collection of solicited information): Data collected only for rostering and timesheet purposes.
  • APP 5 (Notification of the Australian Privacy Principles): Users informed how data is used and stored.
  • APP 6 (Use or disclosure): Data only disclosed to SKS supervisors and payroll admins.
  • APP 11 (Security): Encryption, access controls, and audit logging protect data.
  • APP 12 (Access and correction): Supervisors can view and update employee details.

Data Scope

The following personal and workplace data is stored:

  • Employee names and phone numbers
  • Email addresses
  • Work schedules (site assignments per day)
  • Timesheets (job numbers, hours worked)
  • Leave requests (type, dates, status)
  • Supervisor/manager assignments

Transport Security

The following HTTP security headers are enforced on all responses via Netlify:

HeaderValue / Purpose
HSTSmax-age=31536000; includeSubDomains — forces HTTPS for 1 year
X-Frame-OptionsDENY — prevents clickjacking
X-Content-Type-Optionsnosniff — prevents MIME type sniffing
Referrer-Policystrict-origin-when-cross-origin — limits URL leakage
Permissions-PolicyDisables camera, microphone, payment, USB, FLoC
Content-Security-PolicyRestricts script/style sources to self and required CDNs

CORS Policy

All Netlify Functions validate the request Origin header against a whitelist of approved domains. Only requests from sks-nsw-labour.netlify.app, eq-solves-field.netlify.app, and localhost (development) are accepted. Branch deploys and deploy previews are also permitted via subdomain matching. A Vary: Origin header is set to prevent CDN cache poisoning.

Secret Management

Environment Variable Injection (Implemented v1.1)

All server-side secrets are managed through Netlify environment variables, encrypted at rest and injected at runtime only. No API keys, tokens, or cryptographic salts are required in the codebase.

EQ_SECRET_SALT
HMAC-SHA256 signing key for session tokens (shared across all 3 functions)
AUDIT_SB_URL
Supabase REST API endpoint for audit log writes
AUDIT_SB_KEY
Supabase publishable API key for audit log writes
RESEND_API_KEY
Resend email API key for leave request notifications
ANTHROPIC_API_KEY
Anthropic API key for EQ Agent chat proxy

Completed 13 April 2026: All server-side secrets migrated from hardcoded values to Netlify environment variables. Fallback values have been stripped — functions now fail explicitly with a clear error if any required env var is missing. Zero secrets remain in the codebase.

Remaining Client-Side Codes

Known limitation: Staff access codes (staffCode, supervisorCode) remain in the client-side app-state.js as part of the tenant branding configuration. These are shared access PINs (not secrets in the traditional sense) and are visible to anyone inspecting the page source. Transition to server-side-only validation is planned for v2.0 (per-user credentials).

Additional Measures

  • GitHub secret scanning: Active on the repository. Runs on every push.
  • Private repository: Source code is in a private GitHub repo with restricted access.
  • Netlify scopes: Environment variables scoped to functions context only — not exposed during build or post-processing.
  • Code review: All changes reviewed before merge to main branch.

Remediation Roadmap

This roadmap outlines the planned security hardening of the Field app. The v1.0 release prioritizes functionality and rapid deployment; v2.0 and beyond focus on production-grade hardening.

DONE Secure send-email function (add authentication) No authentication — anyone can call it x-eq-token header validation required; HMAC-SHA256 session token check v1.1 ✓
DONE XSS input sanitisation on dynamic HTML Incomplete escaping in onclick handlers Full HTML entity escaping on all user-sourced values in dynamic markup v1.1 ✓
DONE Add HSTS header No Strict-Transport-Security header max-age=31536000; includeSubDomains on all responses v1.1 ✓

Why This Roadmap?

The Field app was built for rapid initial deployment. As of v1.1 (13 April 2026), all CRITICAL items have been resolved: secrets migrated to env vars (fallbacks stripped), send-email authenticated, CORS whitelisted, XSS sanitised, HSTS enabled, input validation added, and audit logging extended to the AI agent. HIGH items prepare the platform for wider adoption and regulatory scrutiny. MEDIUM items align with mature SaaS practices.

Progress update (v1.1): All CRITICAL items resolved (8 of 8 completed). Zero CRITICAL items remain. HIGH items are scheduled for v2.0 (Q3 2026) assuming no production incidents that demand earlier action.
